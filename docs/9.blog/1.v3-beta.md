# Nitro v3 测试版来啦！

> Nitro v3 现已作为公开测试版发布——这是服务器框架的全新进化，基于 Web 标准、Rolldown、Vite v8 构建，并保持“随处部署”的承诺。

<!-- 👉 已在使用 Nitro v2？查看 [迁移指南](/docs/migration) 。 -->

## 简短回顾

Nitro 最初作为 [Nuxt 3](https://nuxt.com) 的服务器引擎诞生，旨在解决一个特定问题：与部署无关的服务器。
随着时间推移，Nitro 超越了 Nuxt，成为多个元框架的基础及构建独立服务器的工具包。

在 Nitro v3 中，我们借此机会重新思考了基础架构。更精简的 API、Web 标准、一级支持 [Rolldown](https://rolldown.rs/) 和 [Vite v8](https://vite.dev/) 集成，同时带来更佳的用户和代理体验（稍后详述！）

自从我们在首届 [Vite Conf](https://viteconf.amsterdam/)（2025年10月11日）低调发布 v3 [alpha.0](https://github.com/nitrojs/nitro/releases/tag/v3.0.1-alpha.0) 起，Nitro v3 已被许多用户采用（周下载量约达 [~28万](https://npmtrends.com/nitro-vs-nitro-nightly)！）并通过精彩的贡献和反馈不断完善，包括 [Tanstack Start](https://tanstack.com/start/latest/docs/framework/react/guide/hosting#nitro)、[Vercel Workflows](https://useworkflow.dev/docs/getting-started) 以及如 [T3Chat](https://t3.chat/) 等生产应用。

特别感谢 VoidZero（Vite 和 Rolldown）、Nuxt（[v5 即将发布！](#nuxt-v5)）、TanStack Start 团队以及所有帮助 Nitro v3 达到这一里程碑的贡献者。❤️

## 为什么要构建服务器？

我们不会将原始源文件发送到浏览器。我们使用构建工具，因为它们解决了实际问题：**HMR（热模块替换）** 实现即时反馈，**代码拆分** 只加载路由需要的部分，**摇树优化** 消除无用代码，**压缩** 以减小负载体积。Webpack 和随后出现的 [Vite](https://vite.dev/) 将前端开发从痛苦转变为高效。

但前端应用并非孤立存在，它们需要 API、数据库、认证、实时数据。它们需要服务器。

随着无服务器和边缘计算兴起，服务器端正面临多年前前端解决的同样限制。**冷启动** 意味着启动的每一毫秒都很重要。**内存限制** 很严格——臃肿的依赖可能超标。**包体积** 直接影响部署速度和启动时间。且你的代码需要运行在各处：Node.js、Deno、Bun、Cloudflare Workers、Vercel 等。然而多数服务器框架仍然发布未优化、未打包的代码，假设长期运行进程，在这种环境下上述问题无关紧要。

Nitro 将构建工具理念带到后端。你可以期待与前端工具一样出色的开发体验：快速迭代的 HMR、由 Rolldown 驱动的优化构建以及摇树优化后的生产环境输出，性能近乎裸机。**单一代码库，支持任意运行时，任意平台。**

## ⚡ 一级 Vite 集成

Nitro 现有原生 [Vite](https://vite.dev) 插件，用于构建全栈应用。

```ts [vite.config.ts]
import { defineConfig } from "vite";
import { nitro } from "nitro/vite";

export default defineConfig({
  plugins: [nitro()],
});
```

在 Vite 应用中添加 `nitro()` 即可获得：

- 通过文件系统路由实现的 **API 路由**
- 与前端构建集成的 **服务器端渲染**
- 一个生产服务器——一次 `vite build` 即生成包含前后端的优化 `.output/` 文件夹，准备部署

这意味着你可以为任何 Vite 项目添加完整后端——参见 [示例](/examples)，包含 [React](/examples/vite-ssr-react)、[Vue](/examples/vite-ssr-vue-router) 和 [Solid.js](/examples/vite-ssr-solid)。

## 🚀 默认性能至上，零冗余

Nitro 在构建时编译路由。运行时没有路由器——每个路由按需加载。仅加载执行特定请求所需代码。

使用 `standard` 预设构建的最小服务器包体积不到 `10kB`，可通过 [srvx](https://srvx.h3.dev/) 以接近原生速度提供服务，并包含了来自 [H3](https://h3.dev/) 的所有优秀功能。

我们还大幅减少依赖数量，从之前的 [321 个依赖](https://npmgraph.js.org/?q=nitropack) 减少到 [不到 20 个](https://npmgraph.js.org/?q=nitro-nightly)。

## 🖌️ 新标识：`nitro`

Nitro v3 以全新 NPM 包名发布：[`nitro`](https://npmx.dev/package/nitro)，取代旧的 `nitropack`。

所有导入均使用简洁的 `nitro/*` 子路径：

```ts
import { defineNitroConfig } from "nitro/config";
import { defineHandler } from "nitro";
import { useStorage } from "nitro/storage";
import { useDatabase } from "nitro/database";
```

不再使用深层 `nitropack/runtime/*` 路径，同时你可以在构建器外导入 nitro 子路径，方便单元测试。

## 🔧 自由选择框架

Nitro v3 对你的 HTTP 层不做约束。你可以使用内置的文件系统路由，也可以通过 `server.ts` 入口文件完全掌控，自由选择你喜欢的框架：

```ts [server.ts]
import { Hono } from "hono";

const app = new Hono();
app.get("/", (c) => c.text("Hello from Hono!"));

export default app;
```

## 🌐 搭载基于 Web 标准的 H3 (v2)

Nitro v3 升级到 [H3 v2](https://h3.dev)，它已完全重写，基于 Web 标准原语：[`Request`](https://developer.mozilla.org/en-US/docs/Web/API/Request)、[`Response`](https://developer.mozilla.org/en-US/docs/Web/API/Response)、[`Headers`](https://developer.mozilla.org/en-US/docs/Web/API/Headers) 和 [`URL`](https://developer.mozilla.org/en-US/docs/Web/API/URL)。

结果是更简洁、更便携的服务器代码：

```ts [routes/hello.ts]
import { defineHandler } from "nitro";

export default defineHandler((event) => {
  const ua = event.req.headers.get("user-agent");
  return { message: "Hello Nitro v3!", ua };
});
```

读取请求体使用原生 API：

```ts [routes/submit.ts]
import { defineHandler } from "nitro";

export default defineHandler(async (event) => {
  const body = await event.req.json();
  return { received: body };
});
```

无多余封装，也不抽象平台已提供的功能。如果你了解 Web API，便能理解 H3 v2。

[Elysia](https://elysiajs.com/)、[h3](https://h3.dev)、[Hono](https://hono.dev)——任何支持 Web 标准的框架均可与 Nitro 配合使用。

## 🗄️ 内置原语

Nitro 搭载了强大且体积小的、**完全可选的** 运行时不可知服务器原语，支持所有运行时。

::note
未使用时，不会增加额外服务器包体积。你仍可在 Nitro 内置原语旁使用平台原生原语。
我们也在为开发环境提供平台特定原语的一级模拟支持，详见 [env-runner](https://github.com/unjs/env-runner) 和 [nitrojs/nitro#4088](https://github.com/nitrojs/nitro/pull/4088)。
::

### 存储

运行时无关的键值层，支持 20+ 驱动——文件系统、Redis、S3、Cloudflare KV、Vercel Blob 等等。可为不同命名空间附加驱动，无需改动应用代码即可切换。

```ts
import { useStorage } from "nitro/storage";

const storage = useStorage();
await storage.setItem("user:1", { name: "Nitro" });
```

:read-more{to="/docs/storage"}

### 缓存

为服务器路由和函数缓存功能，基于存储层。内置支持 stale-while-revalidate、TTL 和自定义缓存键。

```ts
import { defineCachedHandler } from "nitro/cache";

export default defineCachedHandler((event) => {
  return "我已缓存一小时";
}, { maxAge: 60 * 60 });
```

:read-more{to="/docs/cache"}

### 数据库

内置 SQL 数据库，开发时默认 SQLite，也支持 Postgres、MySQL 及更多数据库，通过统一 API 连接。

```ts
import { useDatabase } from "nitro/database";

const db = useDatabase();
const users = await db.sql`SELECT * FROM users`;
```

:read-more{to="/docs/database"}

## 🌍 随处部署

将服务器构建成优化的 `.output/` 文件夹，兼容：

- **运行时**：Node.js、Bun、Deno
- **平台**：Cloudflare Workers、Netlify、Vercel、AWS Lambda、Azure、Firebase、Deno Deploy 等

无需配置—Nitro 会自动检测部署目标。利用平台特性如 ISR、SWR 和边缘渲染，代码一行不改。

## 🎨 服务器端渲染

使用你喜欢的模板引擎渲染 HTML，或直接在服务器使用 React、Vue、Svelte 等组件库。实现完整的通用渲染与客户端水合。

Nitro 打下基础，采用渐进式方法——先使用 API 路由，需要时添加渲染，按需扩展到完整 SSR。

## 🔄 从 v2 迁移

Nitro v3 引入了破坏性变更以奠定更清晰基础，主要包括：

- `nitropack` → `nitro`（包重命名）
- `nitropack/runtime/*` → `nitro/*`（简洁子路径导入）
- `eventHandler` → `defineHandler`（H3 v2）
- `createError` → `HTTPError`（H3 v2）
- Web 标准的 `event.req` 头和体 API
- Node.js 最低版本：**20**
- 预设名称调整和合并（如 `cloudflare` → `cloudflare_module`）

完整列表见[迁移指南](/docs/migration)。
